# Contributing to pylopdf

Thank you for helping improve pylopdf. Contributions are welcome for PDF
correctness, extraction quality, rendering fidelity, editing behavior,
performance, documentation, packaging, and ecosystem interoperability.

## Report an issue

Search existing issues first and test with the latest pylopdf release when
possible. A useful report includes:

- the pylopdf, Python, and operating-system versions;
- a minimal Python reproducer;
- the expected and actual behavior, including warnings and tracebacks;
- a minimal PDF or a script that generates it.

PDFs frequently contain private information, signatures, attachments, and
metadata. Remove confidential data before sharing a file. Only attach a PDF
that you may legally redistribute, and state its source and license when it is
not generated by the reproducer.

Report security vulnerabilities privately through
[GitHub Security Advisories](https://github.com/yhay81/pylopdf/security/advisories/new)
instead of opening a public issue.

## Development setup

pylopdf is a Python library backed by a Rust extension. The supported
development workflow uses [uv](https://docs.astral.sh/uv/):

```powershell
uv sync --all-extras --group interop
uv run pytest
uv run ruff check .
uv run ruff format --check .
uv run mypy src tests
cargo fmt --manifest-path rust/Cargo.toml --check
cargo clippy --manifest-path rust/Cargo.toml --all-targets -- -D warnings
```

Rust behavior is tested through the public Python API in `tests/`; do not add
Rust unit tests. Add focused synthetic fixtures to regular test modules.
Real-world PDF regressions belong in `tests/test_real_world.py`. Record the
source, license, and known limitations of any bundled file in
`tests/assets/real_world/README.md`.

The optional interoperability tests require:

```powershell
uv sync --all-extras --group interop
uv run pytest tests/test_interop.py
```

## Pull requests

Keep each pull request focused on one coherent behavior. Before submitting:

1. Add or update Python tests for behavior changes.
2. Update `CHANGELOG.md` for user-visible changes.
3. Update the English documentation when the public API or behavior changes.
   English is canonical; translations may follow in the same or a later change.
4. Run the relevant commands above.
5. Include measurements for performance changes, publishing wins and losses
   together.

Commit messages, code comments, docstrings, repository documentation, and
user-facing messages must be in English. Localized documentation and fixtures
needed to test Unicode or CJK behavior are the exceptions.

Small, reviewable changes are easiest to validate. For large features, open an
issue first so the API and corpus requirements can be discussed before
implementation.
